# NVFP4 / W4A4 External Evidence

Status: public primary sources checked 2026-07-16.

This note separates published results from inferences that still require local measurement. In particular, `W4A4` describes four-bit weights and activations for the operators selected by a quantization recipe; it does not necessarily mean that every operator in a model runs in FP4.

## Terminology warning: Gemma `A4B`

In a Gemma model name such as `Gemma-4-26B-A4B`, **`A4B` means approximately 4 billion active parameters**, not four-bit activations. Activation precision must be established separately from the checkpoint's quantization configuration. NVIDIA's Gemma NVFP4 card describes an `nvfp4_experts_only` recipe that quantizes the weights and activations of routed experts; the model name itself does not establish W4A4.

## Bottom line

1. The recalled claims are different: Gemma's published maximum is about **1.44x measured / 1.5x rounded versus BF16**, while Qwen3.6 27B supplies the **2.5x versus NVIDIA's quantized checkpoint** result.
2. Both claims are high-concurrency output throughput. Neither is an isolated prefill benchmark. Unsloth does not publish the input/output-length controls or separate prefill timing needed to treat either headline as a prefill result.
3. A larger W4A4 benefit during compute-bound, high-M prefill is technically plausible, but that is an inference. It should be tested with fixed-length prefill microbenchmarks and end-to-end TTFT, separately from decode and aggregate server throughput.
4. NVIDIA's DeepSeek V4 NVFP4 checkpoint applies W4A4 to routed MoE expert linear operators. Attention, routing, normalization, embeddings, and other unselected operations are not made W4A4 merely by using that checkpoint.
5. An abliterated or otherwise modified DeepSeek MXFP4 checkpoint is not automatically NVIDIA-style NVFP4. The ModelOpt weight cast must be run and verified, and activation amax calibration must supply valid `input_scale` values.
6. The currently deployed abliterated DSpark checkpoint is specifically native-MXFP4/W4A16 for routed experts; its NVFP4 DS-MLA cache setting does not make the expert GEMMs W4A4.
7. Native DeepSeek V4 NVFP4 support is now merged upstream, while the maintained
   SM121 preview uses FlashInfer CUTLASS and reports only a marginal prefill
   advantage over MXFP4. Therefore B12X is an optimization candidate, not the
   assumed production answer. The exact GX10 matrix must compare CUTLASS and
   B12X with identical weights/routes/inputs before either is promoted.
8. NVIDIA's own TensorRT-LLM backend already uses a phase crossover: CUTLASS
   at large M and B12X at small M. Our prefill plan must first measure that
   crossover on GB10; forcing B12X for every M can make TTFT substantially
   worse even though decode improves.
9. What remains specific to this deployment is the hybrid checkpoint contract:
   NVIDIA NVFP4 target layers plus native-MXFP4 DSpark draft layers, one shared
   B12X arena if B12X is selected, no silent fallback, correct activation
   scales, numerical quality, chunked-prefill behavior, CUDA-graph stability,
   and single-/multi-node performance.
